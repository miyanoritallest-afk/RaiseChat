# 要件定義書

## 1. プロジェクト概要

### サービス概要

RaiseChatは、チームのコミュニケーションを効率化するSlack風チャットアプリケーション。
ワークスペース・チャンネルという階層構造でチームの会話を整理し、リアルタイムで情報を共有できる。

### 対象ユーザー

- 複数人のチームで日常的に連絡・情報共有を行うユーザー
- ワークスペースを作成してチームを管理するオーナー
- 招待されてワークスペースに参加するメンバー

### 想定利用シナリオ

```
1. オーナーがワークスペースを作成し、招待コードやユーザーIDでメンバーを招待する
2. メンバーはチャンネルに参加し、テキストやファイルを共有しながら会話する
3. 特定の話題はスレッドで返信し、チャンネルのタイムラインを汚さずに議論する
4. 少人数での会話はDMを使い、チャンネルを作るほどでもない連絡を手軽に行う
5. メンションや未読通知で重要な情報を見逃さない
```

## 2. 機能要件

### ① 認証・ユーザー管理

| 機能 | 概要 | 必要な理由 |
|---|---|---|
| 新規登録 | ユーザーID・表示名・パスワードで登録（メール認証不要） | 手軽に始められるよう認証フローをシンプルにする |
| ログイン／ログアウト | JWT認証でセッションを管理 | セキュアな認証を実現する |
| プロフィール設定 | アバター画像・表示名・ステータス・ステータスメッセージ | 誰が発言しているかを視覚的に識別できるようにする |

### ② メッセージング

| 機能 | 概要 | 必要な理由 |
|---|---|---|
| チャンネル内テキストチャット | リアルタイムでメッセージを送受信する | チャットアプリのコア機能 |
| メッセージの編集・削除 | 送信後にメッセージを修正・削除できる | 誤送信に対応できるようにする |
| スレッド（返信）機能 | メッセージに対してスレッドで返信できる | 複数の話題が混在するタイムラインを整理する |
| チャンネルにも投稿するオプション | スレッド返信をチャンネルタイムラインにも流せる | スレッドを追っていないメンバーにも結論を届ける |
| マークダウン記法対応 | コードブロック・太字・リストなどの書式を使える | エンジニアなどのユーザーがコードを共有しやすくする |
| 画像・動画ファイルの添付 | メッセージにファイルを添付できる | テキスト以外の情報も共有できるようにする |
| 絵文字リアクション | メッセージに絵文字でリアクションできる | 返信するほどでもない共感・承認を手軽に表現する |
| メンション（@ユーザー名） | 特定のユーザーに通知を送れる | 重要なメッセージを見逃さないようにする |
| メッセージ検索 | キーワードでメッセージを検索できる | 過去の会話を素早く見つけられるようにする |
| メッセージのピン留め | 重要なメッセージをチャンネル全員が参照できる場所に留める | いつでも見返せる重要情報の置き場所を作る |

### ③ ワークスペース・チャンネル管理

| 機能 | 概要 | 必要な理由 |
|---|---|---|
| ワークスペースの新規作成 | 独立したチーム環境を作成できる | 複数のチーム・プロジェクトを切り替えて使えるようにする |
| #general自動生成 | WS作成時に全員が参加するデフォルトチャンネルを自動で作成 | 全体への連絡場所を確保する |
| 参加時のシステムメッセージ | ユーザーがWSに参加した際に#generalに通知を流す | 新メンバーの参加をチーム全員が把握できるようにする |
| パブリックチャンネル | ワークスペースのメンバーなら誰でも参加可能なチャンネル | オープンな情報共有を促進する |
| プライベートチャンネル | 招待されたメンバーのみが参加できるチャンネル | 機密性の高い会話の場所を確保する |
| 友人の招待 | 招待コードまたはユーザーIDでメンバーを招待できる | チームにメンバーを追加する手段を提供する |

### ④ 管理者権限（オーナー）

| 機能 | 概要 | 必要な理由 |
|---|---|---|
| ユーザーの招待 | ユーザーIDで招待できる | オーナーが主体的にチームを構成できるようにする |
| ユーザーのキック | メンバーを強制退出させられる | 不適切なユーザーへの対処手段を提供する |
| チャンネルの追加・削除 | チャンネルの管理権限を持つ | チーム構成に合わせたチャンネル整理を可能にする |

### ⑤ ダイレクトメッセージ

| 機能 | 概要 | 必要な理由 |
|---|---|---|
| 1対1 DM | 特定の1人とプライベートに会話できる | チャンネルを作るほどでもない個別連絡の場所を提供する |
| グループDM | 複数人（2人以上）でプライベートに会話できる | チャンネルを立てるほどでもない少人数の会話を手軽にする |

### ⑥ 通知

| 通知種別 | タイミング | 必要な理由 |
|---|---|---|
| メンション通知 | @ユーザー名でメンションされたとき | 自分宛のメッセージを見逃さないようにする |
| 未読メッセージ通知 | 未読メッセージがあるとき | チャンネルの活動状況を把握できるようにする |
| チャンネル招待通知 | プライベートチャンネルに招待されたとき | 招待に気づけるようにする |
| ワークスペース招待通知 | ワークスペースに招待されたとき | 招待に気づけるようにする |
| スレッド返信通知 | 自分のメッセージにスレッド返信がついたとき | 自分が発信した話題への反応を見逃さないようにする |
| リアクション通知 | 自分のメッセージにリアクションがついたとき | 自分のメッセージへの反応を把握できるようにする |

## 3. 非機能要件

### パフォーマンス

| 項目 | 目標値 | 補足 |
|---|---|---|
| メッセージ送信のレイテンシ | 500ms以内 | Socket.ioによるリアルタイム配信で実現 |
| メッセージ一覧の初回表示 | 1秒以内 | カーソルベースページネーション（50件）で対応 |
| 検索レスポンス | 2秒以内 | PostgreSQLのインデックスで対応 |

### セキュリティ

| 項目 | 対策 |
|---|---|
| パスワード管理 | bcryptでハッシュ化してDBに保存（生パスワードは保存しない） |
| 認証 | JWTトークンによるステートレス認証 |
| ファイルアクセス制御 | AWS S3のバケットポリシーで直接アクセスを制限 |
| プライベートチャンネル | サーバーサイドで権限チェックを行い、未参加ユーザーはアクセス不可 |
| オーナー操作 | キック・チャンネル削除等はサーバーサイドで権限チェック |

### スケーラビリティ

現スコープはポートフォリオ・個人開発を想定しており、大規模スケールは対象外。
ただし将来的な拡張ポイントは把握しており、必要に応じて以下で対応可能。

- メッセージ取得の高速化 → Redisキャッシュの導入
- 同時接続数の増加 → Redis PubSubによるSocket.ioのスケールアウト
- DB負荷の分散 → リードレプリカの追加

### 可用性

| 環境 | サービス | 制約 |
|---|---|---|
| フロントエンド | Vercel（無料プラン） | 商用利用制限あり・帯域制限あり |
| バックエンド | Render（無料プラン） | 15分間アクセスがないとスリープする |
| DB | Render PostgreSQL（無料プラン） | 容量1GB・90日で削除される可能性あり |

> ポートフォリオ用途として許容範囲とする。本番運用時は有料プランへの移行が必要。

## 4. 制約条件

| 制約 | 内容 |
|---|---|
| 開発リソース | 個人開発・学習目的 |

## 5. 拡張候補（スコープ外）

### キャッシュ機能（Redis）

大規模利用時のパフォーマンス改善として検討。現スコープでは実装しない。

| 用途 | 概要 |
|---|---|
| メッセージ一覧のキャッシュ | 頻繁に取得されるチャンネルの最新メッセージをRedisに保存しDB負荷を軽減 |
| オンライン状態の管理 | ユーザーのオンライン/オフライン状態をRedisで高速管理 |
| タイピングインジケーターの状態管理 | 一時的な「入力中」状態をRedisで管理（DBに保存するほどでない） |
| セッション管理 | ログアウト済みJWTトークンのブラックリストをRedisで高速管理 |
